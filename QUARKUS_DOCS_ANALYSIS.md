# Quarkus Documentation vs. Reality: ObjectMapperCustomizer with Reflection-Free Serializers

## What Quarkus Documentation Claims

From https://quarkus.io/guides/rest#reflection-free-jackson-serialization-and-deserialization:

> **"Developers can further customize JSON processing by implementing the ObjectMapperCustomizer interface. This interface allows fine-grained control over the ObjectMapper, enabling the registration of custom serializers and deserializers while ensuring compatibility with the reflection-free optimization."**

## The Reality: This Is Misleading ⚠️

This statement is **partially false** and creates unrealistic expectations about what ObjectMapperCustomizer can do with reflection-free serializers.

---

## What ACTUALLY Works

Based on our analysis of PR #55216 and the Quarkus codebase:

### ✅ ObjectMapperCustomizer Settings That Work With Reflection-Free

These work because the generated serializers **query them at runtime**:

1. **Visibility settings** (fields, getters, is-getters)
   ```kotlin
   override fun customize(mapper: ObjectMapper) {
       mapper.setVisibility(
           mapper.serializationConfig.defaultVisibilityChecker
               .withGetterVisibility(JsonAutoDetect.Visibility.NONE)
       )
   }
   ```
   **Why it works**: Generated code calls `JacksonMapperUtil.isPublicGetterVisible(serializerProvider)` at runtime

2. **PropertyNamingStrategy**
   ```kotlin
   override fun customize(mapper: ObjectMapper) {
       mapper.propertyNamingStrategy = PropertyNamingStrategies.SNAKE_CASE
   }
   ```
   **Why it works**: Generated code queries `serializerProvider.getConfig().getPropertyNamingStrategy()` at runtime

3. **JsonInclude settings**
   ```kotlin
   override fun customize(mapper: ObjectMapper) {
       mapper.setSerializationInclusion(JsonInclude.Include.NON_NULL)
   }
   ```
   **Why it works**: Generated code checks inclusion at runtime

### ❌ What Does NOT Work With Reflection-Free

Despite what the documentation says, these **fail silently** or are **ignored**:

1. **"Enabling the registration of custom serializers"** ❌
   ```kotlin
   override fun customize(mapper: ObjectMapper) {
       val module = SimpleModule()
       module.addSerializer(MyClass::class.java, MyCustomSerializer())
       mapper.registerModule(module)
   }
   ```
   **Why it fails**: Generated serializers are hardcoded at build time. Your custom serializer is never called!

2. **Jackson Modules (like KotlinModule)** ❌
   ```kotlin
   override fun customize(mapper: ObjectMapper) {
       mapper.registerModule(KotlinModule.Builder().build())
   }
   ```
   **Why it fails**: Modules define behavior that needs to be baked into generated bytecode at build time. Registering at runtime is too late.

3. **Custom Deserializers** ❌
   ```kotlin
   override fun customize(mapper: ObjectMapper) {
       val module = SimpleModule()
       module.addDeserializer(MyClass::class.java, MyCustomDeserializer())
       mapper.registerModule(module)
   }
   ```
   **Why it fails**: Same reason - generated deserializers are already created.

## The Fundamental Architecture Problem

The documentation's claim "**ensuring compatibility with the reflection-free optimization**" is misleading because:

### How Reflection-Free Works
```
BUILD TIME:
├─ Generate StdSerializer implementations
├─ Generate StdDeserializer implementations
├─ Bake serialization logic into bytecode
└─ NO ObjectMapperCustomizer executed yet

RUNTIME:
├─ Application starts
├─ ObjectMapperCustomizer.customize() runs
├─ Registers modules/serializers
└─ But generated serializers don't use them!
```

### What Gets Used
- **Reflection-free path**: Uses generated `YourClass$quarkusjacksonserializer` / `YourClass$quarkusjacksondeserializer`
- **Your custom serializers**: Registered in ObjectMapper but never called for classes with generated serializers
- **Result**: Your customizations are ignored

## Evidence: Your Bug Reports

The bugs you found (Issues #55142, #55214) **prove this documentation is wrong**:

### Issue 1: Duplicate @JsonProperty Fields
- **What happened**: Reflection-free serializers generated duplicate fields
- **Why**: Generated code didn't properly handle @JsonProperty
- **If doc was correct**: ObjectMapperCustomizer could have fixed this → It couldn't!

### Issue 2: Kotlin Boolean "is" Prefix
- **What happened**: Reflection-free serializers applied JavaBean "is" stripping even with KotlinModule
- **Why**: KotlinModule registered via ObjectMapperCustomizer was ignored
- **If doc was correct**: Registering KotlinModule should have worked → It didn't!

### The Fix (PR #55216)
Required **modifying the code generator itself**, not just ObjectMapperCustomizer. This proves the documentation's claim is false.

## What The Documentation SHOULD Say

Here's a more accurate version:

> **"Developers can customize some JSON processing behaviors by implementing the ObjectMapperCustomizer interface. This interface allows control over the ObjectMapper for settings that are checked at runtime, such as visibility, property naming strategy, and inclusion settings. However, custom serializers, deserializers, and Jackson modules registered through ObjectMapperCustomizer will NOT be used for classes with generated reflection-free serializers. For complex customization needs, consider disabling reflection-free serializers with `quarkus.rest.jackson.optimization.enable-reflection-free-serializers=false`."**

---

## What You Should Do

### If You Need Custom Serializers/Deserializers
**Disable reflection-free serializers:**
```properties
quarkus.rest.jackson.optimization.enable-reflection-free-serializers=false
```

### If You Only Need Basic Customization
These work fine with reflection-free enabled:
- ✅ Visibility settings
- ✅ PropertyNamingStrategy
- ✅ JsonInclude settings
- ✅ Date formatting
- ✅ Views
- ✅ Security/Roles

### If You Use Kotlin
- ⚠️ **Test thoroughly** with reflection-free enabled (PR #55216 helps but isn't complete)
- OR **disable reflection-free** for guaranteed compatibility

## Bottom Line

**The Quarkus documentation oversells ObjectMapperCustomizer's capabilities with reflection-free serializers.**

Reality:
- ✅ You can customize **some** settings (visibility, naming, includes)
- ❌ You CANNOT register custom serializers/deserializers effectively
- ❌ You CANNOT rely on Jackson Modules registered via ObjectMapperCustomizer
- ⚠️ The phrase "ensuring compatibility with the reflection-free optimization" is misleading at best

Your bug findings and the need for PR #55216 prove this documentation claim is inaccurate.
