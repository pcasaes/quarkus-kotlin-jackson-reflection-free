# ObjectMapperCustomizer Compatibility with Reflection-Free Jackson

## Customizers That Work With Reflection-Free ✅

These are safe because they're **checked at runtime**:

### 1. **Visibility Settings** ✅
```kotlin
override fun customize(objectMapper: ObjectMapper) {
    objectMapper.setVisibility(
        objectMapper.serializationConfig.defaultVisibilityChecker
            .withFieldVisibility(JsonAutoDetect.Visibility.ANY)
            .withGetterVisibility(JsonAutoDetect.Visibility.NONE)
    )
}
```
**Why it works**: Generated serializers query `SerializerProvider` at runtime

### 2. **JsonInclude Settings** ✅
```kotlin
override fun customize(objectMapper: ObjectMapper) {
    objectMapper.setSerializationInclusion(JsonInclude.Include.NON_NULL)
}
```
**Why it works**: Generated serializers check include settings at runtime

### 3. **Date Formatting** ✅
```kotlin
override fun customize(objectMapper: ObjectMapper) {
    objectMapper.disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS)
}
```
**Why it works**: Generated serializers delegate to ObjectMapper for dates

### 4. **PropertyNamingStrategy** ✅
```kotlin
override fun customize(objectMapper: ObjectMapper) {
    objectMapper.propertyNamingStrategy = PropertyNamingStrategies.SNAKE_CASE
}
```
**Why it works**: Generated serializers query naming strategy at runtime

## Customizers That DON'T Work With Reflection-Free ❌

### 1. **Registering Jackson Modules** ❌
```kotlin
override fun customize(objectMapper: ObjectMapper) {
    objectMapper.registerModule(KotlinModule.Builder().build())
    objectMapper.registerModule(JavaTimeModule())
}
```
**Why it fails**: Modules define serialization logic that needs to be in generated bytecode

### 2. **Custom Serializers/Deserializers** ❌
```kotlin
override fun customize(objectMapper: ObjectMapper) {
    val module = SimpleModule()
    module.addSerializer(MyClass::class.java, MyCustomSerializer())
    objectMapper.registerModule(module)
}
```
**Why it fails**: Generated code doesn't know about custom serializers

### 3. **Custom AnnotationIntrospector** ❌
```kotlin
override fun customize(objectMapper: ObjectMapper) {
    objectMapper.setAnnotationIntrospector(MyCustomIntrospector())
}
```
**Why it fails**: Introspectors run at build time in reflection-based mode, but generated serializers don't use them

## Decision Matrix

| What You're Customizing | Reflection-Free Compatible? | Recommendation |
|------------------------|---------------------------|----------------|
| Visibility (fields/getters) | ✅ YES | Safe to use |
| PropertyNamingStrategy | ✅ YES | Safe to use |
| JsonInclude settings | ✅ YES | Safe to use |
| Date/time formatting | ✅ YES | Safe to use |
| Views (`@JsonView`) | ✅ YES | Safe to use |
| Security/Roles | ✅ YES | Safe to use |
| Jackson Modules | ❌ NO | **Disable reflection-free** |
| Custom Serializers | ❌ NO | **Disable reflection-free** |
| AnnotationIntrospector | ❌ NO | **Disable reflection-free** |
| **Kotlin with jackson-module-kotlin** | ⚠️ **PARTIAL** | **Test thoroughly** (PR #55216 helps but not complete) |

## The Timing Problem

```
BUILD TIME (mvn clean install):
├─ Reflection-free serializers are GENERATED
├─ Uses hardcoded JavaBean conventions
├─ No ObjectMapperCustomizer executed
└─ No KotlinModule knowledge

RUNTIME (when app starts):
├─ Quarkus creates ObjectMapper CDI bean
├─ Calls ObjectMapperCustomizer.customize()
├─ Registers KotlinModule
└─ But serializers are already generated - too late!
```

**ObjectMapperCustomizer always runs at runtime**, even with reflection-free serializers. This creates a fundamental incompatibility for customizations that need to affect code generation.

## What PR #55216 Fixed (Partially)

The PR added **runtime checks** for certain settings:
- ✅ **Visibility settings** - checked at runtime via `SerializerProvider`
- ✅ **PropertyNamingStrategy** - queried at runtime
- ⚠️ **Kotlin detection** - added build-time workaround (checks for `kotlin.Metadata` annotation)

But it **cannot** fully integrate modules registered by customizers because:
- Modules define custom serialization logic
- That logic needs to be baked into generated bytecode
- Customizers don't run until runtime

## Bottom Line

**If you rely on any customizer that registers modules or custom serializers, you should avoid reflection-free Jackson.** 

For other customizations (visibility, naming strategy, include settings), reflection-free works fine as of Quarkus PR #55216.
